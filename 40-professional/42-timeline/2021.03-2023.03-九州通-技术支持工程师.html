      河南九州通医药有限公司｜ERP 应用运维工程师（IT 业务合作伙伴 / 数据分析）

      时间：2021 年 3 月–2023 年 3 月（2 年）

      任职概览

      • 职级定位：应用运维工程师（Application Operations Engineer），定位为 IT 部门驻扎在业务一线的 ” 技术特种兵 ”。

      • 核心职责：为集团 200 亿年营收 规模的核心 ERP 体系提供 L2 级技术保障。跳出被动响应的窠臼，通过 数据分析 (SQL) 与 流程治理 (BPM) 主动解决业务阻塞点。

      • 技术关键词：Oracle PL/SQL · BPM 流程引擎 · 业务数据分析 · ITSM 服务管理 · 供应链逻辑

      • 职业价值：这段经历构建了我对企业级业务流（Business Flow） 与数据流（Data Flow） 的深度理解，为后续转型基础设施架构师打下了 ” 懂业务 ” 的坚实基础。

      核心职责边界与能力模型

      1. 核心业务系统连续性保障 (Business Continuity)

      • 全链路业务支撑：负责 Oracle ERP 采购、销售、库存、财务全模块的异常处理。不同于 L1 客服，主要解决数据逻辑错误、接口传输失败、系统配置冲突等需要技术介入的深层问题。

      • 关键期 SLA 交付：在月度财务结账（Month-End Closing）与年度库存盘点期间，担任技术值守负责人，通过快速修数与纠错，确保财务账实相符，业务不中断。

      2. 数据价值挖掘与 SQL 开发 (Data Intelligence)

      • 敏捷数据服务：针对标准报表开发滞后的痛点，利用 PL/SQL 直接从数据库提取高维经营数据（如：特定批次药品的库龄分析、负毛利销售拦截分析），为管理层决策提供 “T+0” 的数据支持。

      • 数据质量守门人：编写数据校验脚本，定期巡检系统中的脏数据（如：无主库存、未核销应收），主动消除潜在的财务合规风险。

      3. 流程引擎治理 (BPM Governance)

      • 审批流逻辑维护：管理集团 BPM 流程引擎的配置，解决复杂的 ” 死循环/断链 ” 问题。深入理解企业内控（SOP） 要求，将业务规则转化为准确的系统审批节点。

      • 流程再造支持：协助业务部门梳理冗余审批节点，通过优化触发规则（如：低金额单据自动审批），提升整体供应链流转效率。

      4. 知识工程与标准化 (Knowledge Engineering)

      • ITSM 体系落地：引入标准化的问题管理流程（接收 →复现 →根因 →修复 →归档），拒绝 ” 野路子 ” 运维。

      • 资产沉淀：建立包含 50+ 个通用场景的 SQL 脚本库 与 故障处理知识库 (KB)，将个人经验转化为团队的可复用资产。

      关键项目实战（精选）

      项目 1：ERP 架构升级（v2→v3）灰度切换与数据清洗

      时间：2022.06–2022.12（7 个月）角色：应用侧技术负责人（一线）

      STAR 深度解析

      S（背景/约束）

      • 架构重构：集团推行 ERP 架构升级，引入多业务实体 (OU) 与 库存组织 (IO) 的财务隔离概念，底层数据结构发生质变（如：客户主数据从全局共享变为组织隔离）。

      • 数据风险：存量历史数据庞大且存在逻辑脏数据，直接迁移极易导致升级后账目不平。

      T（目标/KPI）

      • 确保升级期间业务 ” 零停摆 “，新旧版本财务/库存数据一致性 100%。

      • 协助业务部门平滑过渡到新架构，降低学习成本。

      A（关键行动）

      1. 数据清洗工程：编写复杂的 SQL 关联查询脚本，对迁移前的客户、商品、库存数据进行全量逻辑校验，提前发现并修正 500+ 条历史脏数据（如：库存数量为负、无成本记录），消除了迁移隐患。

      2. 业务逻辑验证：在沙箱环境中，组织业务骨干进行全流程（P2P/O2C）穿测，重点验证新架构下的权限隔离与财务自动记账逻辑是否符合预期。

      3. 超融合保障机制：设计 ” 分级响应通道 “，在切换当周实施 7×24 小时驻点，对因架构变更导致的业务阻断实现 15 分钟内 响应修复。

      R（量化结果）

      • 实现 ERP v3 版本的平滑上线，切换窗口期无重大业务事故（P0 级）。

      • 输出了《v3 版本差异化操作手册》与《常见报错速查表》，被集团作为优秀范本推广至其他省公司。

      项目 2：基于 SQL 的经营数据分析与敏捷报表支持

      时间：2021.08–2023.03（持续性工作）角色：数据分析支持

      STAR 深度解析

      S（背景/约束）

      • 需求堰塞湖：业务部门需求频繁且多变（如：突发要求查 ” 某类特定批号的滞销原因 ”），IT 开发团队排期长，标准报表无法满足敏捷决策需求。

      • 口径离散：不同部门对 ” 销售额 “（发货口径 vs 开票口径）定义不一致，导致数据打架，信任度降低。

      T（目标/KPI）

      • 将临时复杂取数需求的响应时间缩短至 1 小时内。

      • 统一数据提取口径，建立可复用的 参数化 SQL 脚本库。

      A（关键行动）

      1. 建立口径确认机制：充当 ” 业务翻译官 “，在提取数据前，强制与业务方确认统计维度（如：退货是否剔除、赠品是否计入、含税/不含税），消除歧义。

      2. 脚本库建设 (Automation)：沉淀了一套包含 50+ 个常用场景的 PL/SQL 脚本库（覆盖销售排行、库存库龄、应收账款账龄等），实现 ” 输入参数，立等可取 ”。

      3. 赋能 BI 开发：将验证过的 SQL 逻辑提供给 BI 团队（帆软报表），作为标准数据源，协助实现了核心管理报表的自动化。

      R（量化结果）

      • 累计响应 500+ 次复杂数据查询需求，平均交付时间缩短 70%。

      • 这种 ” 敏捷取数 ” 能力帮助财务部门在月结前多次发现因业务误操作导致的金额差异，成功规避了潜在的财务审计风险。

      项目 3：BPM 审批流程诊断与业务逻辑优化

      时间：2021.08–2023.03（持续性工作）角色：流程运维负责人

      STAR 深度解析

      S（背景/约束）

      • 流程黑盒：ERP 审批流程配置复杂，涉及金额、部门、职务多重判定，常出现 ” 审批人为空 ” 或 ” 死循环 ” 导致单据卡死，业务怨声载道。

      • 效率瓶颈：部分低风险业务（如内领耗材）审批链条过长，影响效率。

      T（目标/KPI）

      • 消除流程配置导致的业务卡顿，实现流程故障 30 分钟内 修复。

      • 配合内控部门优化审批逻辑，提升流转效率。

      A（关键行动）

      1. 根因诊断：深入 BPM 引擎后台，通过分析流程日志（Token Flow），精准定位断链原因（如：人员离职导致节点真空、条件表达式逻辑互斥）。

      2. 逻辑优化：梳理全集团审批矩阵，优化节点触发规则。例如：针对低金额标准业务设置 ” 自动审批 ” 规则，针对特殊商品设置 ” 强制风控 ” 节点。

      3. 预防性维护：建立 ” 流程变更模拟测试 ” 机制，所有流程调整必须先在测试环境跑通闭环，杜绝直接在生产环境试错。

      R（量化结果）

      • 累计解决 80+ 起流程阻塞故障，恢复了数百张业务单据的正常流转。

      • 通过流程优化，使得常规办公类审批的平均流转时长缩短了 20%。

      职业转折点 (The Turning Point: Application to Infrastructure)

      为什么从应用运维转型为基础设施工程师？

      1. 痛点驱动 (Push Factor)：厌倦了低效的 ” 救火 ”

        • 混乱的源头：在 ERP 运维后期，我深刻感受到单纯的应用层维护是在 ” 治标不治本 “。集团发布流程的不规范、环境配置的漂移、缺乏统一的 CI/CD 标准，导致我们每天都在重复处理因部署错误引发的故障。

        • 成长的天花板：日复一日的重复性配置与手动 ” 救火 ” 工作，让我意识到自己在应用层的技术成长已趋于饱和。我不想再做一个被动的 ” 报错接收器 “，而是想成为能掌控系统稳定性的 ” 规则制定者 ”。

      2. 愿景驱动 (Pull Factor)：渴望从根源解决问题

        • 架构思维觉醒：我意识到，要彻底终结这种混乱，必须向下沉淀——去建设标准化的基础设施，去实施自动化运维，去构建平台工程。只有把地基打牢，上面的应用才能跑得稳。

        • 转型契机：恰逢部门组织架构调整，基础设施团队急需懂业务逻辑的人来推进 ” 运维开发一体化 ”。

      3. 行动 (Action)：

        • 我主动申请内部转岗，带着对 ERP 业务痛点的深刻理解，投身于 虚拟化、容器化、自动化 (Ansible) 与 可观测性 (Zabbix/ELK) 的建设中，致力于打造一个让应用运维不再 ” 救火 ” 的稳定底座。（衔接至下一段经历：高级系统工程师）

      技能矩阵 (Skill Matrix)

      领域核心技能栈
      应用系统Oracle ERP (P2P/O2C 模块深度理解), BPM Workflow 引擎配置, WMS 逻辑理解
      数据工程Oracle PL/SQL (存储过程/复杂查询), Excel (VBA/Pivot), FineReport (基础)
      方法论ITSM (事件/问题管理), SOP 标准化, 知识库管理 (KMS)
      业务领域财务核算逻辑 (AP/AR/GL/月结), 供应链管理 (进销存逻辑), 医药 GSP 合规

      核心竞争力总结 (Transition Value)

      这段经历如何赋能我现在的 ” 高级系统工程师 ” 角色？

      1. 透视上层应用：不同于只懂服务器的运维，我深入理解跑在服务器上的业务逻辑与数据流向。这让我在做基础设施架构设计时，能更精准地评估业务对 IO、并发和一致性的需求。

      2. 数据驱动思维：两年的 SQL 数据分析经验，培养了我 ” 用数据说话 ” 的习惯。在后来的监控告警治理、容量规划中，我依然沿用这种数据驱动的分析方法。

      3. 流程化基因：从 BPM 流程治理中习得的标准化与内控思维，直接转化为我后来推行 ” 自动化运维 ” 与 ” 零信任审计 ” 时的制度设计能力。
